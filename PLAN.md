# Isekai Dev Dashboard: Comprehensive Implementation Plan

## 1. Background & Motivation
The current `isekai-dashboard` provides a visually appealing, read-only grid layout using Textual. To evolve it into a daily driver for an ML Engineer, it must transition from displaying hardcoded dummy data to being fully stateful, interactive, and AI-assisted. 

We will achieve this by introducing a local SQLite database for state management and an OpenRouter integration for free, on-demand LLM assistance (e.g., generating mnemonics for Japanese study).

## 2. Architecture & Tech Stack Updates
- **Database:** `sqlite3` (Built-in Python standard library). Perfect for local, single-user TUI apps. No background daemons required.
- **LLM Client:** `openai` (Python SDK). We will configure the base URL to point to OpenRouter's API, allowing us to use high-quality free models like `google/gemini-2.5-pro` or `meta-llama/llama-3-8b-instruct:free`.
- **UI Framework:** `textual`. We will heavily utilize Textual's `@work` decorator for async API/DB calls, `ListView` for selectable lists, and `Input` for adding data.

## 3. Database Schema Design (SQLite)
We will create a local file named `isekai.db` with the following schema:

```sql
-- Tracks To-Do items
CREATE TABLE IF NOT EXISTS quests (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    status TEXT DEFAULT 'pending', -- 'pending' or 'done'
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Spaced Repetition System (SRS) for Kana/Vocab
CREATE TABLE IF NOT EXISTS kana_srs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    word TEXT NOT NULL UNIQUE,
    meaning TEXT NOT NULL,
    mnemonic TEXT, -- Generated by OpenRouter LLM
    level INTEGER DEFAULT 0, -- SRS Level (0 = new, 1 = learning, 2 = known, etc.)
    next_review TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```
*(An initialization script will seed the `kana_srs` table from your existing `vocab_katakana.json` file).*

### SQLite Threading Note
Because we will use Textual's `@work(thread=True)` for async database calls, we must manage SQLite's threading restrictions. Either each thread must create its own connection, or we initialize a single connection with `check_same_thread=False` and handle locking for concurrent writes.

## 4. OpenRouter LLM Integration Strategy
To prevent UI freezing, the LLM integration will run entirely in background threads.

1. **API Setup:** The user will provide an OpenRouter API key via an `.env` file or environment variable (`OPENROUTER_API_KEY`).
2. **Auto-Mnemonic Generation:**
   - When a Kana card is drawn for review and it lacks a mnemonic, the dashboard will fire an async request to OpenRouter.
   - **Prompt Example:** *"You are a helpful Japanese tutor for an ML Engineer. Create a short, clever mnemonic to help remember the Katakana word '{word}' which means '{meaning}'. Relate it to software engineering or machine learning if possible. Keep it under 20 words."*
   - The result will be saved back to the `kana_srs` table and displayed on the card's "back" face.
3. **Error Handling:** If the LLM API request fails or times out, the app will degrade gracefully, displaying a "Failed to fetch mnemonic" message to allow the user to continue reviewing the flashcard without blocking the UI.

## 5. UI Interactivity & Component Refactoring

### A. The "Daily Quests" Widget (To-Do List)
- **Current State:** Hardcoded `Label`s.
- **Target State:** 
  - A Textual `ListView` containing `ListItem`s.
  - An `Input` field docked at the bottom to type and add new quests.
  - **Interactions:** Pressing `Enter` or clicking a `ListItem` will toggle its status in the SQLite DB and cross it out visually (strikethrough CSS).

### B. The "Kana of the Day" Widget (SRS Flashcards)
- **Current State:** Static text displaying word and meaning simultaneously.
- **Target State:** A stateful "Card" component.
  - **State 1 (Front):** Displays only the large Japanese text.
  - **State 2 (Back):** Triggered by pressing `Space`. Reveals the meaning, romaji, and the LLM-generated mnemonic.
  - **Action Menu:** Once flipped, the footer displays options: `[1] Miss` (Resets SRS level), `[2] Good` (Increases SRS level and delays next review date). Updates the SQLite DB accordingly.

### C. The "Pull Requests" Widget
- **Interactivity:** Upgrade the list to a Textual `ListView`. When a user focuses on a PR and hits `Enter` or `o`, the Python `webbrowser` module (or `gh pr view <number> --web`) will open the PR in the default browser.

### D. The "Calendar" Widget
- **Current State:** Fully functional, fetching data via `icalBuddy` and rendering `Label`s.
- **Target State:** Upgrade to use Textual `ListView` to maintain visual consistency with the other modules, enabling scrollability and potential future interactivity.

## 6. Codebase Refactoring Plan
To ensure the app scales effectively, the current monolithic `main.py` will be split into a structured domain-driven layout:

```text
isekai-dashboard/
├── main.py                  # Entry point (App initialization and Grid layout)
├── db_init.py               # Database seed script
├── core/
│   ├── db.py                # Database connection manager and helper functions
│   └── models.py            # Simple data classes/types (Quest, KanaCard)
├── services/
│   ├── quests_service.py    # DB interactions for Quests
│   └── srs_service.py       # DB interactions for SRS logic
├── clients/
│   ├── llm_client.py        # OpenRouter API integration
│   ├── github_client.py     # gh CLI wrapper
│   └── calendar_client.py   # icalBuddy wrapper
└── ui/
    ├── widgets/
    │   ├── daily_quests.py
    │   ├── kana_card.py
    │   ├── pull_requests.py
    │   └── calendar.py
    └── styles.tcss          # Extracted Textual CSS
```

## 7. Phased Implementation Steps

- [x] **Phase 1: Database Setup.** Write the DB initialization script, define the schema, and migrate data from `vocab_katakana.json` into the `kana_srs` SQLite table.
- [ ] **Phase 1.5: Structural Refactor.** Break `main.py` down into the `core/`, `services/`, `clients/`, and `ui/` module structure before adding new features.
- [ ] **Phase 2: Quest Module Overhaul.** Rewrite the `DailyQuests` class to use Textual's `ListView` and `Input`, directly linked to DB queries via `quests_service.py`.
- [ ] **Phase 3: OpenRouter API Integration.** Add the `openai` dependency via `uv`. Create `llm_client.py` to fetch mnemonics using a free OpenRouter model, handling errors gracefully.
- [ ] **Phase 4: Kana SRS Module Overhaul.** Rewrite the `KanaOfTheDay` widget to implement the "Flip" state machine, integrate the LLM mnemonic fetcher, and handle the spaced repetition logic (calculating `next_review` timestamps).
- [ ] **Phase 5: UX Polish.** Add `webbrowser` hooks for GitHub PRs, convert the PR and Calendar widgets to use `ListView`, refine the CSS styling (moving to `styles.tcss`), and ensure keyboard navigation works seamlessly across the grid.

## 8. Migration & Rollback
- All state changes are confined to the local `isekai.db` file.
- If the database becomes corrupted, the user can delete `isekai.db` and the app will re-initialize it cleanly using the JSON seed file.
- We will lock the Textual dependency version in `uv.lock` to prevent UI regressions.
